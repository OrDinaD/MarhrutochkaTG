# Работа с Beta веткой

## Описание
Бета ветка создана для безопасного тестирования новых функций и изменений перед их внедрением в основную ветку (main).

## Структура веток
- **main** - стабильная продакшн версия
- **beta** - ветка для тестирования новых функций

## Рабочий процесс

### 1. Переключение на бета ветку для разработки
```bash
git checkout beta
git pull origin beta  # убедиться что у вас последняя версия
```

### 2. Внесение изменений
```bash
# Вносите изменения в коде
git add .
git commit -m "Описание изменений"
git push origin beta
```

### 3. Деплой бета версии
- Настройте отдельный сервер/окружение для бета версии
- Деплойте с бета ветки на тестовый сервер
- Проводите тестирование

### 4. Перенос изменений в main (после тестирования)
```bash
# Переключаемся на main
git checkout main
git pull origin main

# Мержим изменения из beta
git merge beta

# Отправляем обновления в main
git push origin main
```

### 5. Альтернативный способ через Pull Request
1. На GitHub создайте Pull Request из beta в main
2. Проведите код ревью
3. Подтвердите merge

## Полезные команды

### Посмотреть текущую ветку
```bash
git branch
```

### Посмотреть все ветки (включая удаленные)
```bash
git branch -a
```

### Сравнить изменения между ветками
```bash
git diff main..beta
```

### Посмотреть логи только для beta ветки
```bash
git log beta --oneline
```

## Рекомендации

1. **Всегда тестируйте** в beta ветке перед мержем в main
2. **Регулярно синхронизируйте** beta с main чтобы избежать конфликтов:
   ```bash
   git checkout beta
   git merge main
   git push origin beta
   ```
3. **Используйте осмысленные коммит-сообщения** для упрощения отслеживания изменений
4. **Создавайте отдельные конфигурации** для beta и production окружений

## Настройка отдельного деплоя для beta

### Railway (если используете)
1. Создайте новый проект в Railway для beta версии
2. Подключите к beta ветке GitHub репозитория
3. Настройте переменные окружения отдельно для beta

### Другие платформы
- Настройте отдельный сервер/контейнер для beta
- Используйте разные базы данных для beta и production
- Настройте отдельные домены/поддомены (например, beta.yourdomain.com)

## Troubleshooting

### Если нужно отменить последний коммит в beta
```bash
git reset --soft HEAD~1  # отменить коммит, но сохранить изменения
git reset --hard HEAD~1  # отменить коммит и все изменения
```

### Если нужно синхронизировать beta с main
```bash
git checkout beta
git fetch origin
git merge origin/main
git push origin beta
```
